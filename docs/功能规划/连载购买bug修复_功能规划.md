# 连载购买Bug修复 - 功能规划

## 1. 功能概述

### 1.1 问题描述
当前系统存在严重的购买逻辑bug：
- 商家发布文章连载后，用户购买了某一期
- 商家更新同一篇文章发布新一期（期号变化）
- 用户再次访问时，系统只检查 `article_id`，认为用户已购买
- 导致用户无需付费即可查看新一期内容

### 1.2 业务场景
每一期都是独立的内容，需要单独付费购买。用户购买的是"某一期"，而不是"整篇文章的所有期"。

### 1.3 与现有系统关系
- 涉及文章购买模块
- 涉及连载/期次模块
- 需要修改订单表结构和购买判断逻辑

## 3. 数据库设计

### 3.1 表结构修改

#### la_article_order 表（文章购买订单表）

```sql
-- 新增字段
ALTER TABLE `la_article_order` 
ADD COLUMN `issue_no` varchar(20) NOT NULL DEFAULT '' COMMENT '购买时的期号' AFTER `article_id`;

-- 添加索引（可选，用于按期号查询）
ALTER TABLE `la_article_order` 
ADD INDEX `idx_issue_no` (`issue_no`);
```

### 3.2 字段说明
| 字段 | 类型 | 说明 |
|------|------|------|
| issue_no | varchar(20) | 用户购买时文章的期号，如 "2024001" |

## 5. 接口规划

### 5.1 受影响的接口

| 接口路径 | 方法 | 变更说明 |
|----------|------|----------|
| /api/article/detail | GET | 购买状态判断增加期号检查 |
| /api/article/buy | POST | 创建订单时记录期号 |
| /api/article/lists | GET | 列表购买状态增加期号检查 |

### 5.2 接口变更详情

#### 文章详情接口
- **路径**: `/api/article/detail`
- **方法**: GET
- **参数**: 无变化
- **返回值**: 无变化
- **逻辑变更**: 判断 `is_buy` 时同时检查 `article_id` + `issue_no`

#### 购买文章接口
- **路径**: `/api/article/buy`
- **方法**: POST
- **参数**: 无变化（article_id）
- **返回值**: 无变化
- **逻辑变更**: 创建订单时从文章表读取当前 `issue_no` 并记录

## 6. 测试步骤

### 6.1 测试准备
1. 商家账号：创建一个系列，发布文章
2. 用户账号：准备购买

### 6.2 测试流程

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 商家发布文章，期号=001，价格=10元 | 文章创建成功 |
| 2 | 用户A购买文章 | 订单记录：article_id=X, issue_no='001' |
| 3 | 用户A查看文章详情 | is_buy=true，可查看付费内容 |
| 4 | 商家更新文章，期号=002 | 期号变更为002 |
| 5 | 用户A再次查看文章详情 | is_buy=false，不可查看新一期付费内容 |
| 6 | 用户A再次购买 | 订单记录：article_id=X, issue_no='002' |
| 7 | 用户A查看文章详情 | is_buy=true，可查看新一期付费内容 |

### 6.3 边界测试
- 文章无期号（issue_no为空）：按原逻辑处理
- 文章免费（price=0）：无需购买
- 商家查看自己的文章：无需购买

## 8. 风险评估

### 8.1 兼容性
- **历史订单**: 已有的订单 `issue_no` 为空，需要处理
- **解决方案**: `issue_no` 为空时，认为用户购买的是"无期号版本"，保持原逻辑

### 8.2 数据迁移
- 不需要迁移历史数据
- 新订单自动记录期号
